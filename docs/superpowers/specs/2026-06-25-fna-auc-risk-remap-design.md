# FNA AUC 风险分映射与管理员文案调整设计

- 日期：2026-06-25
- 状态：待用户复核
- 主题：将 FNA 五档答案的 AUC 风险分映射从 `0.0 / 0.5 / 0.5 / 0.7 / 1.0` 调整为 `0.0 / 0.3 / 0.5 / 0.7 / 1.0`，并修正文案以避免把风险分误解为单题得分

## 背景

当前系统把 FNA 五档答案同时用于两件事：

1. 判定单题是否答对
2. 为 AUC 计算提供单题风险分

这两件事的统计含义不同，但管理员明细页里将第二类数据直接显示为“恶性评分”，容易让人误解成“这题得了多少分”。现有映射里，`倾向不是癌` 与 `不确定` 也都落在 `0.5`，不符合用户预期的风险顺序。

## 目标

1. 保持 FNA 单题“正确/错误”逻辑不变
2. 将 AUC 风险分映射调整为更符合直觉的五档顺序
3. 将管理员明细页文案改为更清楚的统计术语
4. 保持现有接口结构、导出结构和 AUC 算法不变，尽量缩小改动范围

## 非目标

1. 不重命名后端接口字段，例如 `doctor_malignancy_score`
2. 不调整 AUC 的 pairwise 计算方法
3. 不引入新的数据库字段或迁移
4. 不改动医生端答题页面的选项文本
5. 不将 `不确定` 改判为“部分正确”或“方向正确”

## 已讨论方案

### 方案 A：只改风险分映射

- 仅修改五档答案到风险分的映射
- 不改任何前端展示文案

优点：

- 改动最小
- 对现有代码和测试影响最集中

缺点：

- 用户仍可能把“恶性评分”误解为单题得分
- 已知困惑不会被彻底解决

### 方案 B：改风险分映射，并修正文案

- 修改五档答案到风险分的映射
- 将管理员明细中的“恶性评分”改为“AUC风险分”
- 将“真值”改为“真值(0良/1恶)”

优点：

- 统计口径和界面表达一起修正
- 改动依然小，风险可控
- 能直接回应当前实际困惑

缺点：

- 需要同步更新前端展示和相关测试快照/断言

### 方案 C：改映射、改文案，并增加额外说明

- 在方案 B 的基础上增加 tooltip 或提示文案

优点：

- 最不容易被误解

缺点：

- 当前问题主要集中在明细列名和映射取值，新增说明文字收益有限
- 会让界面多一层解释负担

## 选择

采用方案 B。

理由：

- 用户当前最在意的是“风险分值本身”和“它被误读为单题得分”这两个问题
- 方案 B 能同时解决两者，且不会扩大技术改动面
- 方案 C 的额外说明可以保留为后续可选增强，但不应成为这次改动的前置条件

## 设计决策

### 1. 正确性逻辑保持不变

FNA 二分类方向判对规则继续沿用当前逻辑：

- 真值为 `确定不是癌` 时，`确定不是癌` 与 `倾向不是癌` 判为正确
- 真值为 `确定是癌` 时，`倾向是癌` 与 `确定是癌` 判为正确
- `不确定` 仍不计为正确

这条规则影响：

- `Answer.is_correct`
- attempt 的 `correct` 与 `accuracy`
- 用户整体 `correct` 与 `accuracy`

本次不调整。

### 2. AUC 风险分映射改为新的五档顺序

新的答案到风险分映射如下：

| 答案 | 新风险分 |
|---|---:|
| 确定不是癌 | 0.0 |
| 倾向不是癌 | 0.3 |
| 不确定 | 0.5 |
| 倾向是癌 | 0.7 |
| 确定是癌 | 1.0 |

设计意图：

- `倾向不是癌` 明确低于 `不确定`
- `不确定` 保持在中间位置
- `倾向是癌` 保持高于中线但低于完全肯定
- 五档顺序与临床语义一致

### 3. AUC 算法保持不变

现有 AUC 计算继续使用 pairwise 比较：

- 恶性样本分数高于良性样本分数，记 1
- 两者相等，记 0.5
- 恶性样本分数低于良性样本分数，记 0

因此本次变化只来自风险分映射变化，不来自算法变化。

### 4. 管理员明细页文案调整

管理员判读详情表做两处文案修正：

- `恶性评分` -> `AUC风险分`
- `真值` -> `真值(0良/1恶)`

目的：

- 明确该列是 AUC 计算用的排序分值，而不是单题得分
- 明确 `0/1` 列是二分类真值编码，而不是“本题得了 0 分或 1 分”

### 5. 后端字段与导出结构保持稳定

本次不调整以下结构名称：

- 接口字段 `doctor_malignancy_score`
- 明细里的 `truth_binary`
- CSV 中相关列结构

这样做的原因：

- 可以避免影响已有前端调用和潜在的外部分析脚本
- 将本次改动集中在“值”和“展示文案”，减少兼容性风险

  - 无算法结构调整
  - 结果会因新映射自动变化

- `app/services/admin_user_metrics.py`
  - 无逻辑结构调整
  - 用户整体 AUC 会因新映射自动变化

### 管理员界面

- `web/js/admin_attempt_detail.js`
  - 更新表头文案
  - 保持数据来源不变

### 测试

- `tests/test_fna_binary_scoring.py`
  - 更新风险分映射断言
  - 更新 CSV 导出中的风险分断言

  - 更新 attempt 详情里的风险分集合断言

  - 更新用户整体 AUC 预期值

必要时还需检查其他直接写死 `0.5 / 0.7` 旧口径的测试。

## 行为变化说明

### 不会变化的部分

- 单题“正确/错误”
- attempt 正确率
- 用户整体正确率
- 题目真值标签
- AUC 计算公式

### 会变化的部分

- 单题明细里的 AUC 风险分显示
- attempt AUC
- 用户整体 AUC
- 实时 AUC 曲线
- CSV 导出的风险分数值

### 对历史记录的影响

历史 attempt 和用户整体 AUC 不做数据迁移，而是按新映射自动重算。

这意味着上线后：

- 历史 AUC 数值可能与改动前不同
- 差异是预期行为，因为评分口径发生了调整

## 示例

当一题真值为 `确定是癌`、用户回答 `倾向不是癌` 时：

- 正确性结果：仍为“错误”
- AUC 风险分：由旧口径 `0.5` 改为新口径 `0.3`
- 真值编码：仍为 `1`

这正是本次改动要表达的含义：该答案不是“半对”，而是“偏低恶性风险判断”。

## 测试与验证

实现后应至少完成以下验证：

1. 单元/接口测试
   - 风险映射函数返回 `0.0 / 0.3 / 0.5 / 0.7 / 1.0`
   - 正确性判定逻辑保持不变
   - attempt AUC 结果符合新映射
   - 用户整体 AUC 结果符合新映射
   - CSV 导出包含新风险分

2. 前端校验
   - 管理员 attempt 详情表头显示 `AUC风险分`
   - `真值(0良/1恶)` 文案显示正确
   - 明细页数值与接口返回一致

3. 回归检查
   - 全量 `pytest`
   - 至少一次管理员详情页手工验证

## 实施边界

本设计只覆盖“口径与文案调整”，不包含以下后续增强：

- 为风险分列增加 tooltip
- 在导出文件中新增更可读的解释列
- 在页面中增加“AUC如何计算”的帮助说明
- 改名后端字段或版本化导出 schema

这些增强可以单独作为后续体验优化来做。

## 实现建议

实现时应采用一组原子改动完成：

1. 更新 `app/services/fna.py` 中的风险分映射
2. 更新管理员详情页文案
3. 更新所有受影响测试
4. 跑全量测试并做一次管理员详情页验证

这样可以保证：

- 行为变化集中
- 风险来源单一
- 回滚容易
